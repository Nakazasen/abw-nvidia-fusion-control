# CONTRIBUTOR_ONBOARDING

## Purpose

Onboard human contributors without collapsing governance nuance or audit lineage.

## Ownership

- Owner: Governance Recorder
- Canonicality: Canonical onboarding protocol

## SOURCE OF TRUTH

- Recovery sequence: `04_RECOVERY/RECOVERY_PROTOCOL.md`
- Governance ownership: `01_GOVERNANCE/FILE_OWNERSHIP.md`

## Onboarding Steps

1. Read `00_SYSTEM/SYSTEM_IDENTITY.md` and `00_SYSTEM/PROJECT_PHILOSOPHY.md`.
2. Read `00_SYSTEM/NON_CLAIMS.md` and `00_SYSTEM/ENGINEERING_VALUES.md`.
3. Run recovery order in `04_RECOVERY/RECOVERY_PROTOCOL.md`.
4. Confirm file ownership boundaries in `01_GOVERNANCE/FILE_OWNERSHIP.md`.
5. Confirm active blockers in `03_OPERATIONS/CURRENT_BLOCKERS.md` before proposing changes.

## Contributor Safety Guardrails

Never:
- remove fail/rerun lineage artifacts
- rewrite historical verdicts
- overclaim readiness from bounded tests
- collapse roadmap, snapshot, and handoff into one mirrored narrative

Always:
- preserve append-only chronology where applicable
- use canonical references instead of content mirroring
- update strategic + operational files consistently when making governance changes

## Minimum Evidence Standard For Claims

- claim scope stated explicitly
- repo/commit context recorded
- relevant tests or operational checks cited
- residual risks and non-claims preserved
